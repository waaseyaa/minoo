# WP05 — PHPStan baseline regen

## Objective

Regenerate `phpstan-baseline.neon` if the framework upgrade and the WP02–WP04 fixes introduced any new analysis drift. Confirm `./vendor/bin/phpstan analyse` exits 0.

## Context

- CLAUDE.md gotcha: "After adding new files that call `EntityInterface::get()`, regenerate the baseline with `./vendor/bin/phpstan analyse --generate-baseline phpstan-baseline.neon`. The baseline won't auto-update when new files are added."
- The framework alpha.144–171 likely changed type signatures, deprecated methods, or strengthened return types — any of which can produce baseline drift.

## Branch Strategy

Direct work on `main` (after WP02–WP04). No worktree.

## Subtasks

### T022 [P] — Run `./vendor/bin/phpstan analyse`; capture drift

**Purpose**: See whether the upgrade actually introduced any new errors not in the existing baseline.

**Steps**:
1. Run: `./vendor/bin/phpstan analyse 2>&1 | tee /tmp/phpstan.log`.
2. If exit 0 — no drift. Skip to commit-only-if-changed in T024.
3. If errors, read each one. Decide: legitimate new bug (fix in src/), or framework-driven baseline drift (regenerate baseline).

**Validation**:
- Drift inventory captured.

### T023 [P] — Regenerate baseline if needed

**Purpose**: Update the baseline so CI gate passes.

**Steps**:
1. If T022 found legitimate bugs that should be fixed in source rather than baselined, fix them first (escalate to a follow-up issue if scope grows).
2. Regenerate: `./vendor/bin/phpstan analyse --generate-baseline phpstan-baseline.neon`.
3. Re-run: `./vendor/bin/phpstan analyse` — must exit 0.
4. Diff the baseline: `git diff phpstan-baseline.neon`. Sanity-check the diff:
   - Removed entries: good (errors fixed by upgrade).
   - Added entries: each one should be inspectable and rationalize-able as a framework-driven type tightening.
   - Whole-file removal: suspicious — verify the file still exists.

**Validation**:
- `./vendor/bin/phpstan analyse` exits 0.
- Baseline diff reviewed manually.

### T024 [P] — Commit phpstan-baseline.neon

**Purpose**: Land the baseline change.

**Steps**:
1. If baseline unchanged: nothing to commit; close the WP.
2. If changed: `git add phpstan-baseline.neon`.
3. Commit with message:
   ```
   chore(phpstan): regenerate baseline for alpha.171

   Reflects type changes from waaseyaa/* alpha.143→alpha.171 and the
   WP02/WP03 entity fixes.
   ```

**Validation**:
- `git status` clean for `phpstan-baseline.neon`.

## Definition of Done

- [ ] `./vendor/bin/phpstan analyse` exits 0.
- [ ] Baseline committed if changed.

## Risks

- **Hiding real bugs**: A regenerated baseline can absorb genuine new errors. Mitigation: review the diff before committing.
- **Framework-side errors**: PHPStan reports against `vendor/waaseyaa/*` should be ignored by Minoo's PHPStan config (those belong upstream). Verify the baseline only covers Minoo source.

## Reviewer Guidance

- Spot-check 3–5 entries in the diff to confirm they're framework-driven, not Minoo bugs.
- Confirm baseline doesn't suddenly grow by hundreds of entries (would indicate config drift).
