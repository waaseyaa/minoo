# WP04 — Schema diagnostics + migrations

## Objective

Address any schema drift surfaced by alpha.171's new diagnostics: column-vs-`_data` storage drift (#1309), `BUNDLE_SUBTABLE_MISSING` (#1376), and `ORPHAN_BUNDLE_SUBTABLE` (#1301). Generate migrations as needed; confirm `bin/waaseyaa schema:check` exits clean.

## Context

- alpha.167–171 added active diagnostic detectors for schema drift between `fieldDefinitions` and the actual SQLite schema.
- CLAUDE.md gotcha: "Adding a field to `fieldDefinitions` does not ALTER existing SQLite tables. Run `bin/waaseyaa schema:check` to detect drift, then create a migration."
- Past Minoo work added `community_id` columns to several entity tables (alpha.104 community-scoped tenancy work) — these may be in a state where the column exists but `_data` doesn't, or vice versa.
- Migration tables MUST use the `_data` CLOB schema. Content entities: `{id} INTEGER PRIMARY KEY AUTOINCREMENT, uuid CLOB, bundle CLOB, {label} CLOB, langcode CLOB, _data CLOB`. Config entities: `{id} TEXT PRIMARY KEY, bundle CLOB, langcode CLOB, _data CLOB`. Do NOT create individual columns for fields stored in `_data`.

## Branch Strategy

Direct work on `main` (after WP01). No worktree.

## Subtasks

### T016 — Run `bin/waaseyaa schema:check`; capture output

**Purpose**: Establish what drift exists post-upgrade.

**Steps**:
1. Ensure dev DB is at the latest known good state: `bin/waaseyaa migrate:status` and apply any pending pre-upgrade migrations first.
2. Run: `bin/waaseyaa schema:check 2>&1 | tee /tmp/schema-check.log`.
3. Categorize findings:
   - Missing column / extra column.
   - `BUNDLE_SUBTABLE_MISSING`.
   - `ORPHAN_BUNDLE_SUBTABLE`.
   - Column-vs-`_data` drift.

**Files**: `/tmp/schema-check.log`.

**Validation**:
- Output captured. If exit 0 with no findings, skip ahead to T020 verification.

### T017 — Diagnose column-vs-`_data` drift findings

**Purpose**: For each column-vs-`_data` finding, confirm root cause.

**Steps**:
1. For each finding, check:
   - Is the column the legitimate one (some fields like `community_id`, `uuid`, `bundle`, `langcode` ARE real columns)?
   - Is the field stored in `_data` AND duplicated in a column?
   - Is there an old migration that added the column but never backfilled `_data`?
2. Decide remediation per finding:
   - Column legitimate → update entity definition to declare the column-backed storage.
   - `_data` is the truth, column is stale → drop the column in a new migration.
   - Both are written but column is canonical → add a migration to backfill `_data` from column.

**Validation**:
- Each finding has a documented remediation choice.

### T018 — Diagnose `BUNDLE_SUBTABLE_MISSING` / `ORPHAN_BUNDLE_SUBTABLE` findings

**Purpose**: Resolve bundle-subtable mismatches.

**Steps**:
1. `BUNDLE_SUBTABLE_MISSING` indicates an entity bundle that registered field definitions but has no backing subtable. Likely fix: a migration creating the subtable with the standard `_data`-CLOB schema.
2. `ORPHAN_BUNDLE_SUBTABLE` indicates a leftover subtable for a bundle that no longer exists. Likely fix: drop the subtable in a new migration. Verify the bundle is genuinely retired (check `src/Provider/Entity/*Provider.php` for any registration).

**Validation**:
- Each bundle finding has a confirmed remediation.

### T019 — Generate migrations via `bin/waaseyaa make:migration <name>`

**Purpose**: Codify each remediation as a migration file.

**Steps**:
1. For each remediation from T017 and T018, generate a migration:
   ```
   bin/waaseyaa make:migration <descriptive_name>
   ```
2. Edit the generated PHP file in `migrations/` following the existing pattern. Reference an existing migration for the `_data`-CLOB schema if creating a new content/config entity table.
3. Use `up()` and `down()` symmetrically; `down()` should reverse the structural change.

**Files**: `migrations/<timestamp>_<name>.php` (one per remediation).

**Validation**:
- Each migration file exists and PHP-lints clean (`php -l migrations/...`).

### T020 — Run migrations; re-verify schema:check is clean

**Purpose**: Apply the migrations and confirm drift is gone.

**Steps**:
1. Apply: `bin/waaseyaa migrate`.
2. Verify: `bin/waaseyaa migrate:status` shows new migrations as Up.
3. Re-run: `bin/waaseyaa schema:check`.
4. Expected: exit 0 with no findings.
5. If new findings appear (chained drift), iterate T017–T020.

**Validation**:
- `bin/waaseyaa schema:check` exits 0.
- All new migrations applied in dev DB.

### T021 — Commit migrations

**Purpose**: Land the schema fixes.

**Steps**:
1. `git add migrations/`.
2. Commit with message:
   ```
   chore(schema): resolve alpha.171 schema drift

   Adds migrations for findings reported by the new column-vs-_data,
   BUNDLE_SUBTABLE_MISSING, and ORPHAN_BUNDLE_SUBTABLE diagnostics
   (framework #1309/#1376/#1301).
   ```

**Validation**:
- `git status` clean for migrations/.

## Definition of Done

- [ ] `bin/waaseyaa schema:check` exits 0 with zero findings.
- [ ] Migrations committed.
- [ ] Each migration has a working `down()`.

## Risks

- **Migration body shape**: Wrong schema (individual columns instead of `_data` CLOB) causes "no column named _data" errors at runtime. Mitigation: copy from an existing working migration.
- **Production migration cost**: If a remediation requires backfilling production data, do NOT apply it during the deploy window — flag for a separate maintenance issue and ship the upgrade with a "drift accepted" note.
- **Down migrations**: Some structural changes (dropping a column) can't cleanly reverse without data loss. Document irreversibility in the migration file's docblock.

## Reviewer Guidance

- Confirm each migration uses the `_data` CLOB schema for content/config entities.
- Confirm any column drops are intentional (data was already in `_data`).
- Confirm `down()` is sane.
