# WP01 — Composer bump and resolution

## Objective

Bring every `waaseyaa/*` package from the current alpha.142–143 mix up to alpha.171, with a clean lockfile and passing strict validation.

## Context

- Currently installed: 31 packages at `0.1.0-alpha.143`, 9 at `0.1.0-alpha.142` (notably `api`, `bimaaji`, `entity`, `entity-storage`, `field`, `geo`, `graphql`, `media`, `mercure`, `path`, `relationship`).
- Target: `0.1.0-alpha.171` for all 40 packages.
- The mismatch between `alpha.142` and `alpha.143` indicates a coordinated split lag from the framework monorepo — alpha.171 should resolve all packages to the same version.
- `composer.json` may have `repositories` path entries pointing to `../waaseyaa/packages/{entity,field,genealogy}` for sibling-repo work. These must be addressed (kept, removed, or sibling tree updated).

## Branch Strategy

Direct work on `main`. No worktree allocation needed for this WP — it's a manifest/lockfile change committed in place.

## Subtasks

### T001 — Bump waaseyaa/* constraints in composer.json to ^0.1.0-alpha.171

**Purpose**: Update version constraints so Composer is allowed to resolve to alpha.171.

**Steps**:
1. Read `composer.json`. Locate the `require` and `require-dev` blocks.
2. For every `waaseyaa/*` entry, update the constraint to `^0.1.0-alpha.171` (or `~0.1.0-alpha.171` if stricter pin desired).
3. Preserve any non-waaseyaa constraints unchanged.
4. Do NOT modify `repositories` entries in this subtask — that's T002.

**Files**: `composer.json`.

**Validation**:
- `git diff composer.json` shows only waaseyaa/* version bumps and no other changes.
- `composer validate --no-check-lock --no-check-publish` exits 0 (lockfile mismatch is expected at this stage; will resolve after T003).

### T002 — Decide path-repository question (entity/field/genealogy overrides)

**Purpose**: Make the decision about path-repo overrides explicit so resolution is deterministic.

**Steps**:
1. Inspect `composer.json` `repositories` array. Identify any `type: path` entries pointing to `../waaseyaa/packages/{entity,field,genealogy}`.
2. Check sibling tree state: `cd ../waaseyaa && git log --oneline -5` and check tags. If sibling tree is at or beyond alpha.171, keep the overrides. If behind, decide: (a) update sibling tree to alpha.171 commit, (b) remove the overrides for this upgrade and re-add later, or (c) leave hybrid and document.
3. Document the decision in a one-line comment in `composer.json` above the `repositories` array AND in `kitty-specs/upgrade-waaseyaa-alpha-171-01KQTDC2/research/path-repos-decision.md` (one paragraph max).
4. If overrides removed, also remove any `@dev` constraints in `require` for those packages and replace with `^0.1.0-alpha.171`.

**Files**: `composer.json`, `kitty-specs/upgrade-waaseyaa-alpha-171-01KQTDC2/research/path-repos-decision.md`.

**Validation**:
- Decision is captured in writing.
- `composer validate --no-check-lock --no-check-publish` exits 0.

### T003 — Run `composer update 'waaseyaa/*' --with-dependencies`

**Purpose**: Resolve the lockfile to alpha.171 across the dependency tree.

**Steps**:
1. Backup the existing lockfile: `cp composer.lock composer.lock.pre-upgrade` (kept locally only, not committed — already gitignored if not, add to gitignore for the duration of the WP).
2. Run: `composer update 'waaseyaa/*' --with-dependencies 2>&1 | tee /tmp/composer-update.log`.
3. If the resolver fails with a conflict, read the error carefully — common causes:
   - A path-repo override pinned to an older sibling commit (re-decide T002).
   - A transitive dep of a waaseyaa package conflicting with a Minoo direct require.
   - A platform requirement (PHP version, ext) bumped upstream.
4. Do NOT use `--ignore-platform-reqs` to paper over real platform mismatches; resolve them.

**Files**: `composer.lock`, `/tmp/composer-update.log` (transient).

**Validation**:
- `composer update` exits 0.
- `/tmp/composer-update.log` shows alpha.171 for every waaseyaa package.

### T004 — Run `composer validate --strict` and `composer install --dry-run`

**Purpose**: Confirm the resolved state is internally consistent.

**Steps**:
1. `composer validate --strict` — must exit 0 with no warnings.
2. `composer install --dry-run` — confirms no further changes would be applied; exit 0.

**Validation**:
- Both commands exit 0.
- No warnings about composer.json schema, license, or constraints.

### T005 — Verify all 40 packages at alpha.171

**Purpose**: Final sanity check that no package was left behind.

**Steps**:
1. Run: `composer show 'waaseyaa/*' | awk '{print $2}' | sort -u`.
2. Expected output: a single line `0.1.0-alpha.171`.
3. If multiple versions appear, identify the laggards and re-trigger T003 with explicit constraints.

**Validation**:
- `composer show 'waaseyaa/*' | awk '{print $2}' | sort -u | wc -l` returns `1`.
- That one version is `0.1.0-alpha.171`.

### T006 — Commit composer.json + composer.lock

**Purpose**: Land the dependency change as a single coherent commit.

**Steps**:
1. `git add composer.json composer.lock` (and the path-repos-decision.md research note if created).
2. Commit with message:
   ```
   chore(deps): bump waaseyaa/* to alpha.171

   Resolves all 40 framework packages from the alpha.142/143 mix to
   a unified alpha.171 baseline. See kitty-specs/upgrade-waaseyaa-alpha-171-*
   for the mission spec, plan, and per-WP execution detail.
   ```
3. Do NOT push yet — push happens in WP09 after all downstream WPs land.

**Validation**:
- `git status` is clean except for files owned by other WPs.
- `git log -1 --stat` shows composer.json + composer.lock changed.

## Risks

- **Path-repo lag**: Sibling worktrees at older commits will defeat the upgrade silently. T002 mitigates by forcing an explicit decision.
- **Transitive conflicts**: A non-waaseyaa transitive dep may pin a version that conflicts with what alpha.171 expects (e.g., Symfony component version). Resolve by upgrading the conflicting Minoo direct require, not by force-flagging the resolver.
- **Bimaaji/api/entity at alpha.142**: These were at `.142` while siblings were at `.143` — likely a Packagist split-repo replication lag. After the bump, all should land on `.171`. If any persist below `.171`, file a framework issue.

## Reviewer Guidance

- Confirm the diff in `composer.json` is *only* version bumps + path-repo decision.
- Confirm `composer.lock` is fully resolved (no `dev-main` or `@dev` resolutions for non-path packages).
- Spot-check one or two `vendor/waaseyaa/{pkg}/composer.json` files to confirm the installed version matches the lock entry.
