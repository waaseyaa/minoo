# WP03 — Unit tests + entities: alpha.165 `_data` symmetry

## Objective

Reconcile Minoo's entity classes, access policies, and unit tests with the FieldStorage `_data` read/write symmetry tightening introduced in alpha.165 (Mission #1257). All non-key field values must round-trip through `_data` consistently — direct column access for fields stored in `_data` is no longer tolerated.

## Context

- alpha.165 release notes: "WP04 #1257: read/write symmetry for FieldStorage::Data", "WP05 #1257: `_data` value coercion in query builder".
- CLAUDE.md gotcha: "Entity fields live in `_data` JSON blob: target_type, target_id, user_id, created_at on engagement entities (reaction, comment, follow) are NOT real SQL columns. Raw SQL queries against them fail."
- The framework now actively enforces this — tests that previously worked through accidentally-tolerant code paths may now fail.
- Hot-path entity types: 17 of them, but the engagement entities (reaction, comment, follow), post, and the new community-scoped types (per HasCommunityInterface) are the most likely to surface symmetry failures.

## Branch Strategy

Direct work on `main` (after WP01 lands). No worktree.

## Subtasks

### T011 — Run `--testsuite MinooUnit`; capture failures verbatim

**Purpose**: Establish a diagnostic baseline before fixing anything.

**Steps**:
1. Run: `./vendor/bin/phpunit --testsuite MinooUnit 2>&1 | tee /tmp/unit.log`.
2. Extract failure list: `grep -E "^(FAILURES|Errors|Failed|^[0-9]+\))" /tmp/unit.log`.
3. If exit 0 — surprise! — skip to T014 and verify, then close the WP.
4. If failures, group them by suspected root cause:
   - FieldStorage `_data` symmetry (look for "column not found", "null value for", or assertion mismatches on entity field reads).
   - Query builder `_data` coercion (look for query result shape errors).
   - Other framework drift (note for Phase 2 of T012).

**Files**: `/tmp/unit.log` (transient), inline notes in commit message.

**Validation**:
- Failure inventory captured.

### T012 — Diagnose FieldStorage `_data` symmetry failures

**Purpose**: For each suspected `_data` symmetry failure, confirm the diagnosis by reading the entity class and the failing assertion.

**Steps**:
1. For each entity type implicated in failures, open `src/Entity/<EntityName>.php`. Verify:
   - `entityKeys` declaration is correct.
   - `fieldDefinitions` enumerates every field stored in `_data`.
   - The constructor isn't bypassing field setters in a way that breaks symmetry.
2. For each access policy implicated, open `src/Access/<PolicyName>.php`. Verify entity field reads use `$entity->get('field_name')` not `$entity->field_name`.
3. For each unit test implicated, open the test and verify it sets fields through `set()` and reads through `get()`, not via raw constructor or array access.
4. Capture the "real fix" plan in `kitty-specs/upgrade-waaseyaa-alpha-171-01KQTDC2/research/data-symmetry-fixes.md` (one paragraph per fix).

**Validation**:
- Each failure has a documented root cause and proposed fix.

### T013 — Fix Minoo entity classes / access tests for symmetry

**Purpose**: Apply the fixes from T012.

**Steps**:
1. Apply fixes one entity / one test at a time. Run the failing test in isolation after each fix: `./vendor/bin/phpunit --filter '<TestName>'`.
2. If a fix requires touching `src/Provider/Entity/*Provider`, do so but keep the change scoped — registering a missing field in `EntityType` for symmetry is in scope; rewriting the provider boundaries is NOT.
3. Avoid `final` on services that need mocking (per CLAUDE.md gotcha).
4. If a fix would touch >5 files, pause — that's a signal of broader drift than one WP can absorb. Escalate per the Complexity Tracking section in plan.md.

**Files**: `src/Entity/**`, `src/Access/**`, `src/Provider/Entity/**`, `tests/App/Unit/**`.

**Validation**:
