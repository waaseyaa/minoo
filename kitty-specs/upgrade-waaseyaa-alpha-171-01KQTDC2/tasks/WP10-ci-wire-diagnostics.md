# WP10 — Wire schema diagnostics into CI

## Objective

Add the alpha.171 schema diagnostic checks (`BUNDLE_SUBTABLE_MISSING`, `ORPHAN_BUNDLE_SUBTABLE`, column-vs-`_data` drift) to Minoo's CI workflow so future schema drift fails the build instead of being discovered during the next framework upgrade.

This is the P3 user story (US5) and is OPTIONAL — if WP01–WP09 took longer than expected, spin this out as a follow-up issue rather than expanding the mission.

## Context

- New diagnostics are exposed via `bin/waaseyaa schema:check`.
- Minoo CI currently runs lint, PHPUnit, Playwright, security audit, commercial-use check (per V1 release governance).
- The right place to add this is the existing PHPUnit / migration job, not a brand-new workflow.

## Branch Strategy

Direct work on `main` (after WP09 ships and is verified). No worktree.

## Subtasks

### T044 — Add `bin/waaseyaa schema:check` invocation to CI workflow

**Purpose**: Make the diagnostic command run on every PR and push.

**Steps**:
1. Locate the relevant CI workflow file (probably `.github/workflows/ci.yml` or similar — list with `ls .github/workflows/`).
2. Find the job that already does `composer install` and `php artisan` / `bin/waaseyaa` migrations.
3. Add a step after migrations:
   ```yaml
   - name: Schema drift check
     run: bin/waaseyaa schema:check
   ```
4. Ensure the step's working directory and PHP version match the surrounding steps.

**Files**: `.github/workflows/<workflow>.yml`.

**Validation**:
- YAML lints clean.
- The step runs in the correct job and order.

### T045 — Add CI step asserting diagnostics fail the build

**Purpose**: Make sure schema:check failures actually fail the workflow run (not just print warnings).

**Steps**:
1. `bin/waaseyaa schema:check` should already exit non-zero on findings — verify this by reading the command source or running it against a known-bad state.
2. If exit code is not gated, wrap the step:
   ```yaml
   - name: Schema drift check
     run: |
       bin/waaseyaa schema:check
       # Exit code from schema:check determines pass/fail
   ```
3. Avoid `continue-on-error: true` — that would defeat the purpose.

**Validation**:
- A simulated failure causes the workflow run to fail.

### T046 — Verify CI fails on a deliberate drift

**Purpose**: Prove the gate works end-to-end.

**Steps**:
1. On a throwaway branch: introduce a deliberate drift (e.g., add a field to an entity's `fieldDefinitions` without a migration).
2. Push the branch, open a draft PR.
3. Confirm CI fails with the expected diagnostic output.
4. Revert the deliberate drift.
5. Confirm CI passes again on the revert.
6. Close the draft PR (do not merge).

**Validation**:
- Failed CI run captured (via screenshot or `gh run view <id>` URL in T047 commit message).
- Revert restores green.

### T047 — Commit CI workflow changes

**Purpose**: Land the gate.

**Steps**:
1. `git add .github/workflows/`.
2. Commit:
   ```
   ci: gate schema:check diagnostics

   Adds bin/waaseyaa schema:check as a CI gate to catch BUNDLE_SUBTABLE_MISSING,
   ORPHAN_BUNDLE_SUBTABLE, and column-vs-_data drift before merge. Verified
   end-to-end with a deliberate drift on a throwaway branch (see PR #N).
   ```

**Validation**:
- `git status` clean for `.github/workflows/`.
- Push triggers a CI run; that run is green (no current drift, since WP04 cleared it).

## Definition of Done

- [ ] `bin/waaseyaa schema:check` runs on every CI run.
- [ ] Failed schema:check fails the workflow.
- [ ] Verified end-to-end with a deliberate drift.
- [ ] Workflow change committed.

## Risks

- **CI runtime cost**: schema:check requires booting the kernel against a real DB. Verify it doesn't add more than ~30s to CI runtime.
- **False positives**: If the diagnostic has known noise (e.g., a check that fires on every fresh DB before migrations), gate it appropriately. Don't suppress real errors.

## Reviewer Guidance

- Confirm the step is positioned after migrations, not before.
- Confirm there's no `continue-on-error: true`.
- Confirm the deliberate-drift verification was actually performed (not just claimed).
