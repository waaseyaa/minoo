# WP09 — Push, deploy, verify production

## Objective

Push the upgrade to `main`, watch GitHub Actions auto-deploy complete, and verify production health using body-size + title checks (not just HTTP status). Document the deploy outcome.

## Context

- Per project memory: "Auto-deploys via GitHub Actions on push to main."
- Per project memory: "After deploys, verify body size and title, not just HTTP status. Zero-byte 200s are the classic WSOD signature."
- Per project memory: "When a deploy pipeline has been red for 3+ runs, assume multiple stacked bugs. Each fix exposes the next. Budget 2-4 fix-deploy-verify rounds, not one."
- Production URL: `https://northcloud.one` (Minoo's production hostname).
- The deploy health check for admin session is currently disabled (commented out in `deploy.php`) per CLAUDE.md known issues — don't be alarmed if it's not in the verify list.

## Branch Strategy

Direct push to `main`. No worktree. This is the cutover WP.

## Subtasks

### T039 — Final full PHPUnit run on clean repo

**Purpose**: Last-line verification before pushing.

**Steps**:
1. Confirm working tree is clean: `git status` (only mission-local files unrelated to this WP, if any).
2. Run: `./vendor/bin/phpunit 2>&1 | tail -20`.
3. Expected: exit 0, test count >= 914, assertion count >= 2568, skips <= 3.
4. If failure, STOP. Push must not happen. Re-open whichever WP owns the failing surface.

**Validation**:
- Full test suite green.

### T040 — Push to main

**Purpose**: Trigger the auto-deploy.

**Steps**:
1. Confirm current branch is `main`: `git branch --show-current`.
2. Confirm commits to push: `git log @{u}..HEAD --oneline` shows the WP01–WP08 commits.
3. Push: `git push origin main`.
4. Capture the push timestamp for the deploy log.

**Validation**:
- `git push` succeeds.
- `git status` shows up-to-date with origin.

### T041 — Watch GitHub Actions deploy run

**Purpose**: Confirm the deploy pipeline reaches completion.

**Steps**:
1. Watch: `gh run watch` (or `gh run list --workflow=deploy.yml --limit 1` to find the run, then `gh run watch <id>`).
2. If a step fails, read the logs: `gh run view <id> --log-failed`.
3. If lint or PHPUnit fails in CI but passed locally, that's a CI-environment delta — investigate and fix forward (per project memory: incident layering, expect 2-4 rounds).
4. If deploy step succeeds, verify the deployer logs landed a new release: SSH or check `/home/deployer/minoo/current` symlink target on production.

**Validation**:
- GitHub Actions deploy run completes with success status.

### T042 — Verify production with curl body-size + title checks

**Purpose**: Confirm the upgraded code is actually serving traffic and rendering content.

**Steps**:
1. For each of the 5 canonical routes, run:
   ```
   curl -sS -o /tmp/prod-<slug>.html -w "code=%{http_code} size=%{size_download} url=%{url_effective}\n" https://northcloud.one/<route>
   ```
2. Routes:
   - `/` (public homepage)
   - `/feed` (302 expected for anon; sufficient signal that routing is live)
   - `/games` (games hub)
   - `/communities/sagamok` (community detail with NC adapter)
   - `/elder-support` (Elder Support form)
3. Each must return HTTP 200 (or 302 for /feed) with body > 1KB and `<title>` present where applicable.
4. Spot-check `<title>` content: `grep -o '<title>[^<]*</title>' /tmp/prod-*.html`.
5. If ANY route returns body < 100 bytes with status 200 — production is in WSOD. STOP the verification, dump `/var/log/php-fpm/*.log` from the production server, file an incident.

**Validation**:
- All 5 routes return expected status with body sizes > 1KB.
- `<title>` tags present and contain expected content.

## Definition of Done

- [ ] Full PHPUnit suite green pre-push.

## Risks

- **Stacked deploy failures**: Per project memory, expect 2-4 fix-deploy-verify rounds when the pipeline has been red. Don't despair on the first failure; do diagnose each one to root cause.
- **Production WSOD**: If body sizes are tiny, the upgrade has broken the response emit path. Classic root cause: `Response::send()` regression. Hotfix: revert to the prior commit on `main` and re-investigate locally.
- **NC adapter on production**: Production source-manager runs at `localhost:8050` from the production app's perspective — that's the *production-internal* source-manager, not the public proxy. If `/communities/<slug>` 500s, it may be an NC client config issue, not the framework upgrade per se.

## Reviewer Guidance

- Confirm any incidents are linked to follow-up issues if not resolved in this WP.
